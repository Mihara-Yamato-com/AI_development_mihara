# 1.プロジェクト概要
## 1.1プロジェクト名：GantForge
## 1.2プロジェクトについて

Webアプリケーション（GantForge）を、AI駆動開発（Claude / Claude Code）で再現することを目的とした学もの。大まかな仕様としてはスケジュール管理アプリ。プロジェクトを追加して、スケジュール項目にやるべき作業と期日を追加。そうすれば横軸のスケジュール表に反映され可視化される。他にはプロジェクト概要の編集やマイルストーンの設定、変更履歴、マインドマップやコメント、即時メモ、タスク管理（未着手・進行中・完了）、コメント、スナップショットなどの追加機能がある。またPDF出力やjson形式で出力、データの取り込みなども行うことができる。

---

# 2.技術制約

- 単一のHTMLファイル（`index.html`）のみで完結させる。HTML/CSS/JSをファイル分割しない（`<style>`と`<script>`は`index.html`内にインライン記述する）
- ビルドツール・バンドラー（webpack, vite, npm scriptsなど）は使用しない
- データの永続化にはIndexedDBを使用する（localStorage/sessionStorageは使わない）
- 外部ライブラリ・CDNは原則使用しない（Vanilla JS/CSSのみ）。お題の要件上どうしても必要になった場合のみ、追加前に必ずユーザーに確認する
- レスポンシブ対応は不要。PCのブラウザ表示を前提とする

---

# 3.開発の進め方

- お題の仕様を受け取ったら、実装前に機能を分割し、着手順序を提案してから実装を始める
- データ構造やUI構成など、大きな設計判断は実装前にユーザーに確認する
- 機能ごとに、動作確認できる単位で区切って進める
- 各手順が終わるごとになぜその判断を行いその手順で進めたのかの思考の過程を残してください。

---

# コーディング規約
- インデントは半角スペース2つ
- 変数名はcamelCase、CSSクラス名はkebab-case
- 関数には簡潔なJSDocコメントを付与する
- グローバル変数を避け、IIFEまたはモジュールパターンで名前空間を分離する

---

# 4.Git運用

- ローカルコミット・GitHubへのpushは、必ず事前にユーザーに確認を取ってから実行する。無断でcommit/pushしない

---

# 5.単一ファイル内の構成ルール

- `<style>`と`<script>`は以下の順でセクション分けし、見出しコメントを付ける
  例: /* ==== Section: Gantt Chart Rendering ==== */
- JS内は「初期化 → データ層(IndexedDB) → UI描画 → イベントハンドラ」の順で記述する
- ファイルが肥大化してきた場合、分割ではなく折りたたみ可能なコメントブロックで整理する

---

# 6.データ構造
- IndexedDBのオブジェクトストア構成、各フィールドの型・意味は本ファイルの付録またはdocs/schema.mdに記載し、変更時は必ず更新する

---
# 7.動作確認方法
- 自動テストは導入しない。各機能実装後はブラウザで手動確認する手順を簡潔に示す
- IndexedDBの中身はDevToolsのApplicationタブで確認する
- 想定ブラウザはGoogle Chrome最新版のみ

---
# 8.用語集
- マイルストーン：特定日付に紐づく重要な区切り
- スナップショット：ある時点のプロジェクト全体状態の保存データ

---
# 9.出力機能の方針
- PDF出力はブラウザ標準の印刷機能(window.print + @media print CSS)を基本とする
- 外部ライブラリが必要な場合は導入前に必ずユーザーに確認する
